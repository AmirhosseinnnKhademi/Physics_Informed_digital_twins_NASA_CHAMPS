# ✈️ Physics-Informed Digital Twin — Jet Engine Predictive Maintenance

> **NASA C-MAPSS · PyTorch PINN · DVC Pipeline · MLflow + DagsHub · HTML Dashboard**

A Physics-Informed Neural Network (PINN) digital twin of a turbofan jet engine.
Combines LSTM sequence learning with four thermodynamic constraints to predict
Remaining Useful Life (RUL) — physically interpretable, reproducible via DVC,
and fully tracked with MLflow on DagsHub.

---

## 🏆 What Makes This Different

Most CMAPSS projects on Kaggle train an LSTM and report RMSE. This project goes further:

| Feature | Typical Project | This Project |

| Model | Pure data-driven LSTM | Physics-Informed Neural Network |
| Loss function | MSE only | MSE + 4 thermodynamic constraints |

## 🗂️ Dataset — NASA C-MAPSS

**Download from one of these sources:**
- **Kaggle:** https://www.kaggle.com/datasets/behrad3d/nasa-cmaps
- **NASA Official:** https://data.nasa.gov/dataset/cmapss-jet-engine-simulated-data

Place all files in `data/raw/`. We use **FD001** to start:
- Single fault mode: High Pressure Compressor (HPC) degradation
- Single operating condition: sea level
- 100 training engines, 100 test engines, 21 sensors per cycle

### The Three Files and Their Roles

```
train_FD001.txt   ← engines run TO FAILURE
                    you see the full degradation lifecycle
                    used for training + validation split

test_FD001.txt    ← engines STOPPED EARLY at an unknown point
                    you only see the beginning of degradation
                    model must predict RUL at the last observed cycle

RUL_FD001.txt     ← TRUE RUL values for each test engine
                    NEVER seen by the model during training
                    used ONLY for final blind evaluation
                    this is your proof that the model actually works
```

### Dataset Columns

```
1   unit_id          which engine (1–100)
2   cycle            current operating cycle (time axis)
3   op_setting_1     altitude
4   op_setting_2     Mach number
5   op_setting_3     throttle resolver angle
6–26 sensor_1 to sensor_21   physical sensor readings
```

## ⚛️ Physics Layer — The PINN

This is what separates this project from a standard LSTM.
Four thermodynamic laws are encoded as **loss function constraints**.
The model is penalized every time its predictions violate physical reality.

### Feature vs Loss — Why Physics Goes in the Loss Function

| Physics as Feature | Physics as Loss Term |
|---|---|
| A hint the model can ignore | A law the model is punished for violating |
| Suggested guidance | Hard constraint |
| Model may or may not use it | Guaranteed enforcement |

Degradation is a *law* of how engines wear — not an observable input signal.
It must be **enforced**, not suggested.

---

### Equation 1 — Isentropic Compression (2nd Law of Thermodynamics)

Governs the temperature/pressure relationship across the compressor:

```
T2/T1 = (P2/P1) ^ ((γ-1)/γ)

T1  = compressor inlet temp   → sensor_2  (T24)
T2  = compressor outlet temp  → sensor_3  (T30)
P2  = outlet pressure         → sensor_7  (P30)
γ   = 1.4  (ratio of specific heats for air — physical constant)
```

A healthy engine obeys this exactly.
As HPC degrades, actual T2 rises above what physics predicts.
**The deviation between actual and ideal T2 IS the degradation signal.**

```python
def isentropic_loss(T1, T2, pressure_ratio, gamma=1.4):
    T2_ideal  = T1 * (pressure_ratio ** ((gamma - 1) / gamma))
    violation = (T2 - T2_ideal) / T2_ideal
    return torch.mean(violation ** 2)
```

---

### Equation 2 — Compressor Isentropic Efficiency

The single most important health indicator for HPC degradation:

```
η_c = (T2s - T1) / (T2 - T1)

η_c  = isentropic efficiency
       healthy engine:  η_c ≈ 0.88
       failing engine:  η_c → 0.65

T2s  = ideal outlet temperature  (from Equation 1)
T2   = actual outlet temperature (measured by sensor)
T1   = inlet temperature         (measured by sensor)
```

Physics constraints encoded:
1. Efficiency must stay between 0.60 and 1.0 (impossible outside these bounds)
2. Low efficiency must correlate with low RUL — the model cannot predict
   high RUL for a compressor that is thermodynamically failing

```python
def efficiency_loss(T1, T2, T2_ideal, y_pred_rul):
    eta_c                 = (T2_ideal - T1) / (T2 - T1 + 1e-8)
    upper_violation       = torch.relu(eta_c - 1.0)
    lower_violation       = torch.relu(0.6 - eta_c)
    rul_normalized        = y_pred_rul / 150.0
    consistency_violation = torch.relu(rul_normalized - eta_c)
    return (upper_violation.mean()
            + lower_violation.mean()
            + consistency_violation.mean())
```

---

### Equation 3 — Monotonic Degradation (Paris Law — Materials Science)

From the physics of material fatigue and crack propagation in turbine blades:

```
da/dN = C * (ΔK)^m

a    = crack size
N    = number of cycles
C, m = material constants

Consequence: damage always accumulates, never reverses.
Therefore:   RUL(t+1) < RUL(t) — always, without exception.
```

Without this constraint, a pure LSTM can predict an engine getting
*healthier* over time — physically impossible, but the network doesn't know that.

```python
def monotonicity_loss(y_pred_sequence):
    # y_pred_sequence shape: (batch, time_steps)
    differences = y_pred_sequence[:, 1:] - y_pred_sequence[:, :-1]
    violations  = torch.relu(differences)   # penalize any RUL increase
    return violations.mean()
```

---

### Equation 4 — Energy Balance (1st Law of Thermodynamics)

Work done by the compressor must equal the enthalpy change of the airflow:

```
W_compressor = ṁ * Cp * (T2 - T1)

ṁ    = mass flow rate  (related to fuel flow ratio sensor φ)
Cp   = 1005 J/kg·K     (specific heat of air — physical constant)
T2   = compressor outlet temperature
T1   = compressor inlet temperature
```

As the engine degrades, more work is required for the same temperature rise.
Deviations from the expected work-enthalpy balance signal inefficiency.

```python
def energy_balance_loss(T1, T2, fuel_flow_ratio, N1):
    Cp         = 1005.0
    mass_flow  = 0.0012 * N1
    W_actual   = mass_flow * Cp * (T2 - T1)
    W_expected = fuel_flow_ratio * 180.0
    imbalance  = torch.abs(W_actual - W_expected) / W_expected
    return imbalance.mean()
```

### Combined PINN Loss

```python
def total_pinn_loss(y_pred, y_true, sensors, cycle_seq,
                    λ1=0.1, λ2=0.15, λ3=0.2, λ4=0.05):

    T1  = sensors[:, :, 1]    # T24 — compressor inlet
    T2  = sensors[:, :, 3]    # T30 — compressor outlet
    P2  = sensors[:, :, 5]    # P30 — outlet pressure
    phi = sensors[:, :, 7]    # fuel flow ratio
    N1  = sensors[:, :, 0]    # fan speed

    pressure_ratio = P2 / 14.7
    T2_ideal       = T1 * (pressure_ratio ** (0.4 / 1.4))

    L_data       = F.mse_loss(y_pred, y_true)
    L_isentropic = isentropic_loss(T1, T2, pressure_ratio)
    L_efficiency = efficiency_loss(T1, T2, T2_ideal, y_pred)
    L_monotonic  = monotonicity_loss(y_pred)
    L_energy     = energy_balance_loss(T1, T2, phi, N1)

    L_total = (L_data
               + λ1 * L_isentropic
               + λ2 * L_efficiency
               + λ3 * L_monotonic
               + λ4 * L_energy)

    return L_total, {
        'data':       L_data.item(),
        'isentropic': L_isentropic.item(),
        'efficiency': L_efficiency.item(),
        'monotonic':  L_monotonic.item(),
        'energy':     L_energy.item()
    }
```

## ✅ Evaluation — Blind Test Against NASA Ground Truth

### Why This Matters

Validation loss during training only proves the model fits the training distribution.
The only real proof a model works is performance on **data it has never seen**,
scored against **ground truth it never had access to**.

NASA provides exactly this via `RUL_FD001.txt`.

### Evaluation Flow

```
TRAINING PHASE                        TEST PHASE
─────────────────────────────────     ─────────────────────────────────

train_FD001.txt                       test_FD001.txt
      ↓                                     ↓
  80/20 split                      Last 30-cycle window
  training / validation             per test engine (blind)
      ↓                                     ↓
  val_rmse per epoch              Model predicts RUL
  logged to MLflow                        ↓
  (step=epoch → charts)          Compare vs RUL_FD001.txt
                                   (never seen during training)
                                          ↓
                                  test_rmse, test_mae,
                                  nasa_score → MLflow
                                  (no step= → final numbers)
                                          ↓
                                   predictions.json
                                  (y_true + y_pred included)
```

### NASA PHM08 Scoring Function

The official metric from the NASA PHM 2008 competition.
Asymmetric by design — penalizes late predictions more than early ones:

```python
def nasa_score(y_true, y_pred):
    """
    Predict failure BEFORE it happens → mild penalty
    (unnecessary maintenance, but the engine is safe)

    Predict failure AFTER it happens → severe penalty
    (engine could fail in operation — dangerous)

    Score closer to 0 = better.
    """
    d = y_pred - y_true    # positive = predicted too late
                            # negative = predicted too early
    score = np.where(
        d < 0,
        np.exp(-d / 13) - 1,   # early: gentler exponential
        np.exp( d / 10) - 1    # late:  harsher exponential
    )
    return float(np.sum(score))
```

### Evaluation Implementation

```python
def evaluate_on_test_set(model, X_test, y_true_rul):
    model.eval()
    with torch.no_grad():
        y_pred = model(X_test).squeeze().numpy()

    results = {
        "test_rmse":  float(np.sqrt(mean_squared_error(y_true_rul, y_pred))),
        "test_mae":   float(mean_absolute_error(y_true_rul, y_pred)),
        "nasa_score": nasa_score(y_true_rul, y_pred),
        "n_engines":  len(y_true_rul)
    }

    # Save for DVC metrics
    with open("metrics/test_metrics.json", "w") as f:
        json.dump(results, f, indent=2)

    # Log to MLflow — no step= (final definitive numbers)
    mlflow.log_metrics(results)

    return results, y_pred
```

## 🏗️ Build Order — Step by Step

```
Step 1 — data_pipeline.py     (~1 day)
          Load raw → drop useless sensors → compute RUL labels
          → sliding windows of 30 cycles → normalize → save
          → verify output shape: (n_samples, 30, 14)

Step 2 — physics.py           (~half day)
          Code all 4 equations as standalone functions
          → unit test each with known physical values
          → verify η_c ≈ 0.88 for healthy engine inputs

Step 3 — model.py             (~half day)
          LSTM architecture in PyTorch
          → input (batch, 30, 14) → output (batch, 1)
          → verify gradient flow

Step 4 — pinn_loss.py         (~1 day)
          Combine MSE + 4 physics terms
          → verify each term produces non-zero gradients
          → test monotonicity penalizes increasing RUL sequences

Step 5 — train.py             (~1 day)
          Full training loop → MLflow run
          → log params + per-epoch metrics with step=
          → save best model → metrics/train_metrics.json

Step 6 — predict.py           (~half day)
          Load model + test_FD001.txt
          → predict RUL for all 100 test engines
          → load RUL_FD001.txt → compute test_rmse, test_mae, nasa_score
          → log final metrics to MLflow (no step=)
          → write predictions.json + metrics/test_metrics.json

Step 7 — Wire dashboard       (~2 hours)
          Add fetch('predictions.json') to dashboard/index.html
          → replace simulated JS values with real model output
```
